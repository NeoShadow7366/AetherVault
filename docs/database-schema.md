# Database Schema Reference

## Overview
The AetherVault relies on a single, local SQLite database (`.backend/metadata.sqlite`) to store everything from cached API payloads to your entire generation history. The database operates in **WAL (Write-Ahead Logging)** mode by default to ensure read/write safety when background scanners are operating concurrently with the web UI.

Because the system is designed to be fully local, it uses five distinct tables to securely track state.

---

## 1. `models`
Tracks the files physical presence inside `Global_Vault/` and associates them with CivitAI / HuggingFace metadata.

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | INTEGER | Primary Key |
| `filename` | TEXT | The basename of the model file on disk. |
| `vault_category` | TEXT | Physical category directory (e.g., `Checkpoints`, `LoRAs`). |
| `file_hash` | TEXT (UNIQUE) | SHA-256 hash used universally as the model's primary identifier. |
| `metadata_json` | TEXT | Full JSON payload returned from CivitAI API. |
| `thumbnail_path` | TEXT | Relative path to the downloaded preview image. |
| `last_scanned` | TIMESTAMP | When the background worker last checked this file. |
| `update_available` | INTEGER | Flag (0 or 1) indicating if a newer version exists on CivitAI. |
| `latest_version_id` | INTEGER | The target CivitAI version ID for an OTA upgrade. |

*- An index `idx_hash` accelerates large vault lookups.*

---

## 2. `generations` 
Backs the "My Creations" Gallery, storing the exact parameters required to restore a canvas.

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | INTEGER | Primary Key |
| `image_path` | TEXT | Relative path to the generated output image. |
| `prompt` | TEXT | The positive prompt used. |
| `negative` | TEXT | The negative prompt used. |
| `model` | TEXT | The base model string provided to the engine. |
| `seed` | INTEGER | Random noise seed used for reproducibility. |
| `steps` | INTEGER | Number of generation iterations. |
| `cfg` | REAL | Classifier Free Guidance scale. |
| `sampler` | TEXT | Sampler/Scheduler name (e.g., `Euler a`). |
| `width` / `height` | INTEGER | Final pixel geometry. |
| `rating` | INTEGER | 0-5 star user rating for gallery filtering. |
| `tags` | TEXT | Comma-separated strings for filtering gallery entries. |
| `extra_json` | TEXT | Engine-specific metadata or full node graphs (ComfyUI). |
| `created_at` | TIMESTAMP | Automatic creation timestamp. |

---

## 3. `embeddings`
Powers the local semantic search feature. The background scanner encodes model descriptions into matrices to allow fuzzy intent lookups.

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | INTEGER | Primary Key |
| `file_hash` | TEXT (UNIQUE) | Foreign key relation pointing to the `models` table. |
| `vector_json` | TEXT | The 384-dimensional floating point array generated by `sentence-transformers`. |
| `last_embedded` | TIMESTAMP | High water mark for the background scanner. |

---

## 4. `user_tags`
Allows the user to attach custom tags to models in the vault. 

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | INTEGER | Primary Key |
| `file_hash` | TEXT | Foreign key relation pointing to the `models` table. |
| `tag` | TEXT | The literal string value of the user's tag. |

*- Constrained by a `UNIQUE(file_hash, tag)` index to prevent duplicate tag allocations.*

---

## 5. `prompts`
The Saved Prompt Library. Enables users to stash complicated prompt topologies for quick recall via the Command Palette.

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | INTEGER | Primary Key |
| `title` | TEXT | User-friendly designation for the quick UI panel. |
| `prompt` | TEXT | The positive prompt string. |
| `negative` | TEXT | The negative prompt string. |
| `model` | TEXT | The target model this prompt is designed for. |
| `tags` | TEXT | Grouping tags. |
| `extra_json` | TEXT | Catch-all for engine specific syntax extensions. |
| `created_at` | TIMESTAMP | Record creation date. |

## Database Configuration

### WAL Mode
The database is initialized with `PRAGMA journal_mode=WAL` during `MetadataDB.__init__()`. WAL (Write-Ahead Logging) enables concurrent readers while a single writer operates, which is critical because:
- The background `vault_crawler.py` writes new model records while the HTTP server reads models for the UI.
- The `embedding_engine.py` writes vector data while the frontend performs semantic searches.

Without WAL mode, these concurrent access patterns would frequently trigger `sqlite3.OperationalError: database is locked`.

### Migration Strategy
The database uses a **try-except `ALTER TABLE`** pattern for backward compatibility. When the schema evolves (e.g., to add `update_available` and `latest_version_id` columns), `_init_db()` issues:

```python
try:
    cursor.execute("ALTER TABLE models ADD COLUMN update_available INTEGER DEFAULT 0")
except sqlite3.OperationalError:
    pass  # Column already exists
```

This ensures existing databases are seamlessly upgraded without data loss, while fresh installations create tables with all columns from the start.

> [!NOTE]
> There is no dedicated migration table or version tracking. Each `ALTER TABLE` is independently idempotent.

### Indexes & Constraints

| Table | Index / Constraint | Definition | Purpose |
|-------|-------------------|------------|---------|
| `models` | `idx_hash` | `CREATE INDEX IF NOT EXISTS idx_hash ON models(file_hash)` | Accelerates vault lookups and CivitAI by-hash queries |
| `models` | `UNIQUE(file_hash)` | Column-level constraint | Prevents duplicate model entries; enables `ON CONFLICT` upserts |
| `user_tags` | `UNIQUE(file_hash, tag)` | Table-level composite constraint | Prevents duplicate tag assignments to the same model |
| `embeddings` | `UNIQUE(file_hash)` | Column-level constraint | One embedding vector per model; enables `ON CONFLICT` upserts |

### Foreign Key Relationships
SQLite does not enforce foreign keys by default (`PRAGMA foreign_keys` is OFF). The `file_hash` columns in `embeddings` and `user_tags` are logically referencing `models.file_hash`, but there is no `REFERENCES` clause — referential integrity is maintained by application logic in `MetadataDB`.

### Prompt Library UI Integration
The `prompts` table feeds the **Command Palette** in the frontend (activated via `Ctrl+K`). When a user saves a prompt, it is stored with a title, positive/negative prompt, target model, tags, and optional engine-specific extensions in `extra_json`. The palette supports substring search across `title` and `prompt` columns via SQL `LIKE` queries.

## Operations
All CRUD operations are orchestrated via the `MetadataDB` Python class (`.backend/metadata_db.py`, 516 lines), employing `sqlite3.Row` dictionaries for clean JSON serialization at the API boundary layer.

### Key Method Reference

| Method | Table | Purpose |
|--------|-------|---------|
| `insert_or_update_model()` | `models` | Upserts by `file_hash` using `ON CONFLICT DO UPDATE` |
| `get_model_by_hash()` | `models` | Single model lookup by SHA-256 hash |
| `get_unpopulated_models()` | `models` | Returns models with `NULL` metadata for CivitAI enrichment |
| `save_generation()` | `generations` | Records a new generation with all canvas parameters |
| `list_generations()` | `generations` | Paginates gallery with sort options (newest, oldest, top_rated) |
| `rate_generation()` | `generations` | Updates star rating for a single generation |
| `save_embedding()` | `embeddings` | Upserts a 384-dim vector for semantic search |
| `get_all_embeddings()` | `embeddings` | Bulk retrieval for cosine similarity matching |
| `add_user_tag()` / `remove_user_tag()` | `user_tags` | Tag management with `INSERT OR IGNORE` |
| `save_prompt()` / `list_prompts()` / `delete_prompt()` | `prompts` | Full CRUD for the Prompt Library |
| `get_dashboard_stats()` | Multiple | Aggregate counts for the dashboard analytics widget |
| `export_models_metadata()` | `models` + `user_tags` | Portable backup manifest generation |
| `import_models_metadata()` | `models` + `user_tags` | Batch upsert from exported manifest with transaction safety |
| `remove_models_by_filenames()` | `models` | Transactional bulk delete with rollback on failure |
